## 핵심 네러티브

배치 자동화 시스템을 운영하는 담당자는 매일 배치 오류 SMS를 받는다. SMS에는 GROUP_ID, RUN_ID, job명 정도만 적혀 있지만, 실제 대응을 위해서는 JOB_ID, 타겟 테이블, 타겟 테이블명까지 모두 알아야 한다. 지금은 SMS를 받은 후 엑셀 스프레드시트를 일일이 검색해야 하는데, 이 과정에 5-10분이 걸린다. 문제는 한 배치가 지연되면 그것을 기다리던 다른 배치들이 연쇄적으로 지연되기 때문에, 이 몇 분의 지체가 전체 일일 업무를 밀어낸다는 것이다. 이 웹앱은 이 프로세스를 완전히 뒤집는다: SMS 텍스트를 복붙하고 두 번 클릭하면 2-3초 내에 필요한 모든 정보가 테이블에 떠오른다. 앱이 시작될 때 배치 정보를 로컬 IndexedDB에 미리 로드해두기 때문에, 여러 오류가 연속으로 발생해도 5분 내에 여러 건을 신속하게 처리할 수 있다. 결과적으로 배치 연쇄 지연을 예방하고, 야간 긴급 호출도 줄일 수 있다.

## 구성 요소

**대상** [명시]  
배치 자동화 시스템의 배치 운영담당자 — 일일 데이터 처리 흐름 전체를 책임지고 배치 오류가 발생하면 신속하게 대응해야 하는 역할

**진짜 문제** [명시]  
표면: SMS 오류 메시지만으로는 배치 상세 정보를 빠르게 파악할 수 없다.  
근본: 현재 엑셀 수동 검색 프로세스가 5-10분이 걸리고, 이 지연이 다른 배치들을 연쇄적으로 밀어낸다. 결과적으로 일일 업무 전체가 밀리고 야간 긴급 호출의 원인이 된다.

**맥락·타이밍** [명시]  
배치 오류는 실시간으로 발생하고, 운영담당자는 오류 알림을 받은 순간부터 빠르게 대응해야 한다. 한 배치의 지연이 후속 배치 전체를 지연시키는 연쇄 구조이기 때문에, 몇 분의 신속성이 전체 일정을 좌우한다.

**핵심 관점·베팅** [명시 + 추론]  
① SMS에 담긴 기본 정보(GROUP_ID, RUN_ID, job명)만으로도 전체 배치 상세 정보를 신속하게 찾을 수 있으면 충분하다고 본다. 로컬 IndexedDB를 사용하면 네트워크 지연 없이 초고속 조회가 가능하다.  
② 불필요한 기능(서버 연동, 권한 관리, 자동 재시작, 로그 저장)은 배제하고 핵심(SMS 파싱 + 로컬 고속 검색)에만 집중한다. 이는 개발 시간을 줄이고 운영 유지보수를 단순하게 한다.

**해결 방향** [명시]  
① SMS 메시지를 텍스트로 복붙하면 정규식으로 자동 파싱하여 GROUP_ID, RUN_ID, SEQ_ID, job명을 추출한다.  
② 추출된 정보를 우선순위(GROUP_ID → RUN_ID → SEQ_ID)에 따라 로컬 IndexedDB에서 검색한다.  
③ 매칭된 배치의 7개 정보(GROUP_ID, RUN_ID, SEQ_ID, job명, JOB_ID, 타겟테이블, 타겟테이블명)를 테이블로 명확하게 표시한다.

**성공의 모습** [명시 + 추론]  
운영담당자가 배치 오류 SMS를 받고, 앱에서 텍스트 두 개의 클릭(파싱, 검색)으로 2-3초 내에 필요한 모든 정보를 확인할 수 있다. 평균 대응 시간이 5-10분에서 5분 이내로 단축되고, 배치 연쇄 지연이 감소하며, 야간 긴급 호출도 줄어든다. 운영담당자는 더 이상 엑셀을 뒤지는 스트레스에서 벗어나 오류 원인 분석과 재처리에 집중할 수 있다.

## Problem Statement

배치 자동화 시스템에서 오류가 발생할 때, 배치 운영담당자는 SMS로 받은 **오류 메시지만으로는** 다음 정보를 빠르게 파악하기 어렵습니다:

- 어떤 배치가 실패했는가?
- 해당 배치의 Job ID, 타겟 테이블, 타겟 테이블명은 무엇인가?

결과적으로 오류 알림을 받은 후 **5분 내에 상세 정보를 조회하기 어렵고**, 운영 대응 속도가 느려집니다.

---

## Solution

**배치 오류 정보 조회 웹앱 (MVP)**은 다음 기능을 제공합니다:

1. **SMS 오류 메시지 파싱**: SMS 텍스트를 복붙하면 자동으로 GROUP_ID, RUN_ID, SEQ_ID, job명을 추출
2. **로컬 배치 정보 DB 조회**: IndexedDB에 저장된 배치 정보에서 추출된 ID로 빠르게 검색
3. **결과 테이블 표시**: 7개 정보(GROUP_ID, RUN_ID, SEQ_ID, job명, JOB_ID, 타겟테이블, 타겟테이블명)를 테이블로 표시
4. **빠른 반복 검색**: DB는 앱 시작 시 자동으로 로드되어, 여러 오류를 5분 내에 연속으로 검색 가능

## User Stories

### Phase 1: MVP (필수, 7/15 완성)

#### SMS 오류 메시지 파싱
1. 배치 운영담당자로서, SMS 오류 메시지를 텍스트 영역에 복붙하고 [파싱] 버튼을 클릭하면, 자동으로 GROUP_ID를 추출할 수 있다. 그래서 매번 수동으로 텍스트를 정렬할 필요가 없다.
2. 배치 운영담당자로서, SMS 오류 메시지에서 RUN_ID를 자동 추출할 수 있다. 그래서 배치 시스템에서 실행 ID를 빠르게 파악할 수 있다.
3. 배치 운영담당자로서, SMS 오류 메시지에서 SEQ_ID를 자동 추출할 수 있다. 그래서 배치 시퀀스 정보를 확인할 수 있다.
4. 배치 운영담당자로서, SMS 오류 메시지에서 job명을 자동 추출할 수 있다. 그래서 실패한 배치의 이름을 명확하게 파악할 수 있다.
5. 배치 운영담당자로서, 파싱된 결과가 읽기 전용으로 표시되어 실수로 수정되지 않는다. 그래서 추출된 정보의 정확성을 보장할 수 있다.
6. 배치 운영담당자로서, 파싱에 실패하면 구체적인 오류 메시지(예: "GROUP_ID를 찾을 수 없습니다")를 받는다. 그래서 SMS 메시지 형식을 다시 확인할 수 있다.
7. 배치 운영담당자로서, 기본 SMS 형식 `[배치자동화]작업에러[YYMMDD][HH:MM][GROUP_ID][RUN_ID][job명]`에 맞지 않는 메시지를 파싱하려 하면, 예상 형식을 안내하는 오류 메시지를 받는다. 그래서 SMS 형식을 올바르게 이해할 수 있다.

#### IndexedDB 배치 정보 DB 조회
8. 배치 운영담당자로서, 앱을 처음 열면 배치 정보 DB가 자동으로 IndexedDB에 로드된다. 그래서 파일을 별도로 선택하거나 업로드할 필요가 없다.
9. 배치 운영담당자로서, 파싱 완료 후 [검색] 버튼을 클릭하면, GROUP_ID로 배치 정보를 검색할 수 있다. 그래서 빠르게 해당 배치의 상세 정보를 조회할 수 있다.
10. 배치 운영담당자로서, GROUP_ID로 검색 결과가 없으면 RUN_ID로 자동 재검색된다. 그래서 여러 우선순위로 안전하게 배치를 찾을 수 있다.
11. 배치 운영담당자로서, RUN_ID로도 결과가 없으면 SEQ_ID로 자동 재검색된다. 그래서 최후의 수단으로 배치 정보를 조회할 수 있다.
12. 배치 운영담당자로서, 한 ID로 여러 행이 매칭되면 모두 테이블로 표시된다. 그래서 모든 가능한 배치 정보를 확인하고 선택할 수 있다.
13. 배치 운영담당자로서, 검색 결과가 없으면 구체적인 오류 메시지(예: "GROUP_ID SDF770, RUN_ID SDFPD771과 일치하는 배치가 없습니다")를 받는다. 그래서 SMS 정보를 다시 확인해야 한다는 것을 알 수 있다.

#### 결과 표시 및 UI
14. 배치 운영담당자로서, 검색 결과 테이블에서 GROUP_ID, RUN_ID, SEQ_ID, job명, JOB_ID, 타겟테이블, 타겟테이블명 7개 정보를 한 번에 볼 수 있다. 그래서 필요한 모든 배치 정보를 5분 내에 파악할 수 있다.
15. 배치 운영담당자로서, 검색 결과 테이블은 명확하고 읽기 쉬운 형식으로 표시된다. 그래서 오류 상황 속에서도 빠르게 정보를 이해할 수 있다.
16. 배치 운영담당자로서, 좌우 분할 레이아웃으로 SMS 입력과 검색 결과를 동시에 볼 수 있다. 그래서 파싱과 검색 결과를 비교하며 작업할 수 있다.

#### 반복 검색 및 상태 관리
17. 배치 운영담당자로서, [초기화] 버튼을 클릭하면 SMS 입력창만 비워진다. 그래서 다음 오류 메시지를 빠르게 검색할 수 있다.
18. 배치 운영담당자로서, SMS 입력이 없으면 [파싱] 버튼이 비활성화된다. 그래서 실수로 빈 메시지를 파싱하지 않는다.
19. 배치 운영담당자로서, 파싱 완료 전에는 [검색] 버튼이 비활성화된다. 그래서 파싱이 완료되지 않으면 검색할 수 없다는 것을 명확히 알 수 있다.
20. 배치 운영담당자로서, [초기화] 버튼은 항상 활성화되어 있다. 그래서 어떤 상태에서든 빠르게 초기화할 수 있다.

#### 데이터베이스 초기화
21. 배치 운영담당자로서, 앱 시작 시 배치 정보 JSON이 IndexedDB로 자동 로드되고, 이후로는 모든 검색이 IndexedDB에서 빠르게 이루어진다. 그래서 네트워크 없이 로컬에서 빠르게 검색할 수 있다.
22. 배치 운영담당자로서, IndexedDB 로드에 실패하면 구체적인 오류 메시지(예: "배치 정보 DB 초기화에 실패했습니다. 페이지를 새로고침해주세요")를 받는다. 그래서 문제 해결 방법을 알 수 있다.

### Phase 1.5: 선택적 기능 (DSX 파일 의존성 분석)

**조건**: Claude 기술 검증 후 "가능하고 간단"이라고 판단된 경우에만 포함

23. 배치 운영담당자로서, DataStage DSX 파일을 업로드하면, 해당 배치의 선후행 관계를 "Job1 → Job2 → Job3" 형태로 추출할 수 있다. 그래서 배치 의존성을 빠르게 파악하고 연쇄 오류를 예방할 수 있다.
24. 배치 운영담당자로서, 검색 결과 옆의 "관련 배치 흐름" 섹션에서 해당 배치의 상위/하위 배치를 확인할 수 있다. 그래서 배치 전체 흐름을 한눈에 파악할 수 있다.
25. 배치 운영담당자로서, DSX 파일 분석 실패 시 구체적인 오류 메시지를 받는다. 그래서 DSX 파일을 다시 확인할 수 있다.

## Implementation Decisions

### 기술 스택

- **프론트엔드**: HTML5 + Vanilla JavaScript (외부 프레임워크 없음)
- **로컬 데이터베이스**: IndexedDB + Dexie.js 라이브러리
- **초기 데이터**: JSON 형식 (엑셀에서 사전 변환)
- **배포**: 정적 HTML 파일 또는 정적 서버

### 데이터 구조

**IndexedDB 테이블: batches**

```javascript
{
  groupId: "SDF770",          // 인덱스 - GROUP ID로 검색
  runId: "SDFPD771",          // 인덱스 - RUN ID로 검색
  seqId: "seq_sfpdmg502",     // 인덱스 - SEQ ID로 검색
  jobName: "FW1차_오늘농사_...",
  jobId: "JOB001",
  targetTable: "TB_EXP",
  targetTableName: "수출입"
}
```

### SMS 파싱 형식

**고정 형식**: `[배치자동화]작업에러[YYMMDD][HH:MM][GROUP_ID][RUN_ID][job명]...`

**정규식으로 추출**:
- `GROUP_ID`: 대괄호 안의 알파벳+숫자 형식
- `RUN_ID`: GROUP_ID 다음의 알파벳+숫자 형식
- `SEQ_ID`: SMS 본문에서 `seq_`로 시작하는 문자열
- `job명`: 마지막 대괄호 안의 텍스트

### UI 레이아웃

**좌우 분할 구조**:
- **좌측**: SMS 입력 영역 + 파싱 버튼 + 초기화 버튼 + 파싱 결과 (읽기 전용)
- **우측**: 검색 버튼 + 검색 결과 테이블

### 검색 로직 (우선순위)

1. `GROUP_ID`로 검색
2. GROUP_ID 결과 없으면 `RUN_ID`로 검색
3. RUN_ID 결과 없으면 `SEQ_ID`로 검색
4. 매칭되는 모든 행을 테이블로 표시

### 파일 구조

```
batch-error-viewer/
├── index.html          # 메인 UI (좌우 분할 레이아웃)
├── app.js              # SMS 파싱 + IndexedDB 검색 로직
├── style.css           # 스타일
├── data.json           # 배치 정보 (엑셀에서 변환)
└── lib/
    └── dexie.min.js    # Dexie.js (IndexedDB 래퍼)
```

### 오류 처리 전략

**파싱 오류**:
- GROUP_ID 미추출: "GROUP_ID를 찾을 수 없습니다. SMS 메시지 형식을 다시 확인해주세요."
- RUN_ID 미추출: "RUN_ID를 찾을 수 없습니다. SMS 메시지 형식을 다시 확인해주세요."
- 기타 파싱 실패: 예상 형식 안내

**검색 오류**:
- 결과 없음: "GROUP_ID SDF770, RUN_ID SDFPD771과 일치하는 배치가 없습니다. SMS 정보를 다시 확인해주세요."
- DB 로드 실패: "배치 정보 DB 초기화에 실패했습니다. 페이지를 새로고침해주세요."

### 버튼 상태 관리

| 버튼 | 초기 상태 | 파싱 완료 후 | SMS 입력 없을 때 |
|------|---------|-----------|----------------|

## Testing Decisions

### 좋은 테스트의 특징

좋은 테스트는 **외부 동작(external behavior)만 검증**하고, 구현 세부사항(implementation details)에 의존하지 않습니다:

- SMS 파싱이 올바른 ID를 추출하는가?
- IndexedDB 검색이 올바른 행을 반환하는가?
- UI가 올바른 결과를 표시하는가?

### 테스트 대상 모듈

1. **SMS 파싱 로직**
   - 정상 SMS 메시지에서 GROUP_ID, RUN_ID, SEQ_ID, job명 추출
   - 불완전한 SMS (GROUP_ID 누락, RUN_ID 누락)에서 오류 발생
   - 예상 형식이 안내되는가?

2. **IndexedDB 검색 로직**
   - GROUP_ID로 검색되는가?
   - GROUP_ID 결과 없을 때 RUN_ID로 자동 재검색되는가?
   - RUN_ID 결과 없을 때 SEQ_ID로 자동 재검색되는가?
   - 여러 행 매칭 시 모두 반환되는가?

3. **UI 상호작용**
   - SMS 입력 후 파싱 버튼 활성화
   - 파싱 완료 후 검색 버튼 활성화
   - 초기화 버튼 클릭 시 SMS 입력창 비워짐
   - 파싱 결과가 읽기 전용으로 표시되는가?
   - 검색 결과가 테이블로 표시되는가?

4. **데이터베이스 초기화**
   - 앱 시작 시 data.json이 IndexedDB로 로드되는가?
   - 중복 로드되지 않는가? (첫 번째만 로드)

### 테스트 방식

- **단위 테스트**: SMS 파싱 함수, 검색 쿼리 함수
- **통합 테스트**: IndexedDB 데이터 로드 + 검색
- **E2E 테스트**: SMS 입력 → 파싱 → 검색 → 결과 표시 전체 흐름

## Out of Scope

### Phase 1에 포함되지 않는 기능

- ❌ **서버 연동**: MVP는 로컬 IndexedDB만 사용 (향후 Phase 2에서 API 연동 가능)
- ❌ **데이터 동기화**: 엑셀이 업데이트되면 앱의 data.json도 수동으로 업데이트 필요
- ❌ **오류 이력 저장**: 검색 결과를 저장하거나 내보내는 기능
- ❌ **결과 복사/내보내기**: 테이블을 복사하거나 CSV/Excel로 내보내는 기능
- ❌ **자동 재시작**: 배치 자동 재시작 기능
- ❌ **권한 관리**: 로그인, 권한 체크
- ❌ **상세 로그 조회**: 배치 상세 로그 표시 (향후 확장)

### Phase 1.5의 조건부 범위

- Phase 1.5 (DSX 파일 의존성 분석)는 Claude 기술 검증 후 "가능하고 간단"이라고 판단된 경우에만 포함
- 검증 결과 "복잡하거나 불가능"이면 건너뜀

### 성공 기준 (데모 시나리오)

**첫 번째 검색:**
1. SMS 오류 메시지 복붙 + [파싱] → GROUP_ID, RUN_ID, SEQ_ID, job명 자동 추출
2. [검색] → IndexedDB에서 빠르게 조회
3. 결과 테이블에 7개 정보 명확하게 표시

**두 번째 검색 (5분 내):**
1. [초기화] → SMS 입력창 비우기
2. 새로운 SMS 복붙 + [파싱] + [검색]
3. DB는 이미 로드되어 있어 빠르게 완료

**오류 처리 검증:**
- 파싱 실패 시 구체적 오류 메시지
- 매칭 결과 없을 시 구체적 오류 메시지

### 초기 데이터 준비 (개발 전 필수)

1. 배치 정보 엑셀 파일 준비
2. JSON 형식으로 변환 (온라인 도구 또는 수동)
3. `data.json` 파일로 프로젝트에 저장
4. **예상 시간**: 30분 ~ 1시간

### DSX 파일 기술 검증 (개발 전)

1. DSX 파일 예시 준비
